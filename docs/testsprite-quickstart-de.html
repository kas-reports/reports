Einleitung

TestSprite positioniert sich als KI-Testagent fuer weitgehend autonome Softwaretests. Laut Dokumentation soll ein kompletter Testzyklus ohne manuelle QA-Arbeit in etwa 10 bis 20 Minuten durchlaufen werden. Der zentrale Gedanke hinter dem MCP-Server ist, dass der KI-Assistent in deiner IDE nicht nur Fragen beantwortet, sondern den gesamten Testablauf direkt aus dem Editor heraus orchestriert: Projekt verstehen, Tests planen, Tests ausfuehren, Ergebnisse zusammenfassen und auf Wunsch sogar Korrekturen vorschlagen.

Der Schnellstart besteht im Kern aus drei Teilen: erstens die Voraussetzungen und die Installation des MCP-Servers, zweitens die eigentliche Konfiguration eines Testlaufs und drittens das Auswerten der erzeugten Artefakte. Fuer deutschsprachige Entwicklerinnen und Entwickler ist vor allem wichtig, dass TestSprite kein klassisches Testframework ist, das man per Hand Case fuer Case schreibt. Stattdessen arbeitet man mit einem hochstufigen Prompt in der IDE und uebergibt der Plattform die Projektumgebung, Zugangsdaten, URLs und idealerweise ein Produktanforderungsdokument.

Voraussetzungen

Bevor du installierst, solltest du drei Dinge vorbereitet haben. Erstens brauchst du eine kompatible IDE mit MCP-Unterstuetzung. Die Dokumentation nennt unter anderem Trae, Cursor, Claude Code, Antigravity und VS Code. Zweitens brauchst du ein TestSprite-Konto. Drittens ist lokal Node.js in Version 22 oder hoeher noetig, weil der MCP-Server ueber npx gestartet wird.

Der naechste Pflichtschritt ist der API-Schluessel. Diesen holst du dir im TestSprite-Dashboard unter den Einstellungen im Bereich API Keys. Dort klickst du auf New API Key und kopierst den generierten Schluessel. Ohne diesen Schritt laesst sich keine der IDE-Integrationen sauber aktivieren.

  1. Im TestSprite-Dashboard anmelden.
  2. Zu API Keys unter den Einstellungen wechseln.
  3. Auf New API Key klicken.
  4. Den API-Key sicher kopieren und fuer die IDE-Konfiguration bereithalten.
Installation in Cursor

Fuer Cursor beschreibt TestSprite zwei Wege. Die schnellste Variante ist der One-Click-Install-Link aus der Dokumentation. Danach traegst du den API-Key ein und kannst direkt loslegen. Wer lieber volle Kontrolle ueber die Konfiguration behalten will, nutzt die manuelle MCP-Konfiguration. Wichtig ist ausserdem der Hinweis zur Sandbox: Wenn Cursor MCP-Tools in einem eingeschraenkten Modus startet, funktionieren die TestSprite-Aktionen nicht vollstaendig. Deshalb sollte der Auto-Run-Modus auf Ask Everytime oder Run Everything gestellt werden.

Die manuelle Cursor-Konfiguration aus der Doku lautet:

{
  "mcpServers": {
    "TestSprite": {
      "command": "npx",
      "args": ["@testsprite/testsprite-mcp@latest"],
      "env": {
        "API_KEY": "your-api-key"
      }
    }
  }
}

Wenn nach dem Speichern der gruene Statuspunkt erscheint und die Tools geladen sind, ist die Installation erfolgreich.

Installation in Claude Code und VS Code

Claude Code nutzt einen terminalbasierten Installationsfluss. Du wechselst zuerst in dein Projektverzeichnis und fuehrst dann den MCP-Add-Befehl aus. Danach kannst du mit claude mcp list pruefen, ob die Verbindung steht. Das wirkt fuer Entwickler angenehm direkt, weil der Status sofort in der Shell sichtbar ist.

cd /path/to/your/project
claude mcp add TestSprite --env API_KEY=your_api_key -- npx @testsprite/testsprite-mcp@latest
claude mcp list
TestSprite: npx @testsprite/testsprite-mcp@latest - ✓ Connected

In VS Code erfolgt die Einrichtung ueber die Command Palette und die MCP-Server-Konfiguration. Dort wird derselbe Server eingetragen, ebenfalls mit dem API-Key in der Umgebungsvariable. Das relevante JSON bleibt unveraendert:

{
  "servers": {
    "testsprite": {
      "command": "npx",
      "args": ["-y", "@testsprite/testsprite-mcp@latest"],
      "env": {
        "API_KEY": "your-api-key"
      }
    }
  },
}

Ein gutes Praxisdetail aus der Dokumentation: Nach der Installation solltest du nicht nur auf Fehlermeldungen achten, sondern bewusst pruefen, ob dein KI-Assistent die TestSprite-Tools wirklich sehen und anbieten kann.

Erster Testlauf

Der eigentliche Schnellstart beginnt damit, dass die Anwendung lokal laeuft. Fuer ein Frontend kann das zum Beispiel ein npm run dev sein; fuer ein Backend etwa node index.js. Danach oeffnest du den Chat deiner IDE und gibst den von TestSprite vorgesehenen Startprompt ein:

Can you test this project with TestSprite?

Laut Dokumentation uebergibt man der KI anschliessend den Rest der noetigen Informationen ueber ein Bootstrap-Tool. Konfiguriert werden muessen mindestens vier Bereiche:

  1. Testing Type: Frontend oder Backend.
  2. Scope: gesamtes Codebase oder nur Code Diff.
  3. Test Account Credentials: falls Login noetig ist, zum Beispiel Benutzername/Passwort, Bearer-Token oder API-Key.
  4. Application URLs: etwa lokale Frontend- und Backend-URLs.

Hinzu kommt ein wichtiger, leicht zu uebersehender Punkt: TestSprite erwartet ein Produktanforderungsdokument. Auch ein unfertiger Entwurf reicht laut Doku, weil die Plattform daraus intern ein normalisiertes PRD erzeugt. Fuer Teams bedeutet das: Je besser das Ausgangsdokument, desto sinnvoller die abgeleiteten Testszenarien.

# For frontend applications (examples)
npm run dev          # Usually runs on port 3000, 5173, or 8080
 
# For backend applications (examples)
node index.js        # Usually runs on port 8000, 3001, or 4000
Frontend: http://localhost:5173
Backend: http://localhost:4000
Ergebnisse und naechste Schritte

Nach einem Lauf legt TestSprite im Projektordner einen eigenen Ergebnisbereich an. Dort findest du unter anderem Konfigurationsdateien, normalisierte PRD-Daten, Testresultate und einen menschenlesbaren Report als Markdown- oder HTML-Datei. Genau dieser Punkt macht den MCP-Ansatz fuer Entwickler interessant: Die Dokumentation verspricht nicht nur rohe Testausgaben, sondern eine strukturierte Rueckgabe mit Fehleranalyse, Kategorien und empfohlenen Korrekturen.

Zusammengefasst ist der Schnellstart fuer deutschsprachige Dev-Teams vor allem dann stark, wenn bereits eine MCP-faehige IDE im Einsatz ist und man schnell vom Prompt zum verwertbaren Testreport kommen will. Der wichtigste Rat aus der Doku ist pragmatisch: Anwendung starten, API-Key sauber hinterlegen, Zugangsdaten und URLs korrekt eintragen, ein PRD hochladen und dann den ersten Lauf ueber den Chat ausloesen. Wenn diese Basis stimmt, soll TestSprite die Testplanung und Ausfuehrung weitgehend autonom uebernehmen.

Fuer das Iterieren nach dem ersten Lauf ist auch der Folgeprompt nuetzlich, den die Dokumentation explizit empfiehlt:

Please fix the codebase based on TestSprite testing results.

Damit wird TestSprite nicht nur zum Testwerkzeug, sondern zu einem Workflow-Baustein fuer Analyse, Fehlerbehebung und erneute Verifikation direkt im Editor.
